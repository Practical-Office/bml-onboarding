/to-tickets

To tickets

Break a plan or spec into tracer-bullet vertical slices — each ticket declares what blocks it — then publish to the tracker and work the frontier.

Why

Demoable slices, explicit dependencies

Horizontal tickets (“all backend,” then “all frontend”) stall validation and swamp context windows. to-tickets cuts narrow paths through every layer so each slice is verifiable on its own.

Blocking edges make parallel work safe: you always know what unblocks next. Wide mechanical refactors are the exception — sequenced expand–contract batches instead of forced vertical slices.

When

Use this skill

  • After to-spec when the build needs more than one PR
  • When a conversation already contains an approved plan worth slicing
  • Before the first implement on a multi-part build

Skip when the spec fits a single ticket — implement directly.

How

Steps

  1. Gather context — conversation, or fetch spec issue/path if passed as argument.
  2. Explore codebase (optional) — glossary, ADRs, prefactor opportunities (“make the change easy, then make the easy change”).
  3. Draft vertical slices — each completes a thin end-to-end path (schema, API, UI, tests); demoable alone; one context window; prefactor tickets first if needed.
  4. Assign blocking edges — list which tickets gate each other; no blockers = can start immediately.
  5. Quiz the human — numbered list with title, blocked-by, and what it delivers; ask about granularity and edges; iterate until approved.
  6. Publish — one issue per ticket (or one file per ticket locally), dependency order, native blocking links where available; ready-for-agent on each; do not close parent spec.
  7. Work the frontier — any ticket whose blockers are done; with implement, one at a time, clear context between.

Inputs / outputs

InputsOutputs
Published spec or plan; optional codebase exploration; human approval on breakdown Set of tickets with titles, acceptance criteria, blocked-by edges, ready-for-agent; parent issue unchanged; ordered frontier for implement

Anti-patterns

What to avoid

  • Horizontal layers — “Database migration ticket” with no user-visible outcome.
  • Oversized slices — Tickets that cannot land in one agent session.
  • Wrong blockers — Ticket B blocked by C when B could start independently.
  • Closing the parent spec — Parent stays open as umbrella reference.
  • Stale paths in tickets — Same rule as specs; behavior descriptions over file lists.
  • Forcing vertical slices on wide refactors — Use expand–contract sequencing instead.

Micro-scenario

Abstract example

Spec #41 covers a onboarding checklist with analytics. /to-tickets proposes:

  1. Expose progress API + stub UI — Blocked by: none. Delivers: API returns empty checklist; UI shell renders.
  2. Render items + emit events — Blocked by: #1. Delivers: user sees checklist; completion fires event.
  3. Operator metrics readout — Blocked by: #2. Delivers: internal page shows completion counts.

Human asks to merge #1 and #2 — agent explains demo risk; human keeps three tickets. Published as #42–#44 with blocking links. Frontier: start #42 with /implement.
