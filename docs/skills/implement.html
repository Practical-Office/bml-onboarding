Implement

Build the work described by a spec or ticket — test at pre-agreed seams, review against standards and spec, then commit.

Why

Close one ticket completely

implement is the execution cap on the Build chain. It turns an unblocked ticket into merged-quality work: behavior at public seams, typecheck and tests green, standards and spec reviewed, changes committed.

It owns tdd and code-review — do not run bare tdd in place of implement for experiment builds.

When

Use this skill

  • Any unblocked ticket from to-tickets frontier
  • Tiny builds skipped straight from experiment issue (optional grill first)
  • Measure-phase work only when instrumentation is broken — still one focused ticket

Clear context between tickets. One implement invocation per ticket.

How

Steps

  1. Read the ticket — what to build, acceptance criteria, blocked-by (should be clear).
  2. Implement with tdd — red → green at pre-agreed seams from spec; one vertical slice per cycle; typecheck and single test files regularly.
  3. Full test suite once — at the end of the ticket, before review.
  4. Run code-review — parallel Standards + Spec axes on the diff since the fixed point; address findings.
  5. Commit — to the current branch with message tied to the ticket.
  6. Stop — do not start the next blocked ticket in the same context; return to frontier.

Inputs / outputs

InputsOutputs
One unblocked ticket or tiny spec; pre-agreed test seams; repo standards docs Implemented behavior; tests at seams; review-passing diff; git commit; ticket ready to close

Anti-patterns

What to avoid

  • Bare tdd without implement — Missing standards/spec review and commit discipline.
  • Testing unconfirmed seams — Couples tests to internals; breaks on refactor.
  • Multi-ticket sessions — Context pollution; acceptance criteria blur.
  • Skipping final suite — Single-file green is not integration green.
  • Commit before code-review — Review is part of the skill, not optional polish.
  • Scope creep — Ticket said “emit event”; implement does not rebuild the dashboard.

Micro-scenario

Abstract example

Ticket #42: “Expose progress API + stub UI.” Seam: public GET /onboarding/progress.

Agent writes failing API test → minimal handler → green. Typecheck. Adds empty UI component wired to endpoint. Single-file tests throughout; full suite once.

/code-review against main: Standards notes a missing error shape; Spec confirms only empty checklist required. Fix applied.

Commit: “#42 expose onboarding progress API and stub UI.” Ticket closed; frontier advances to #43 in a fresh session.

Owned skills

tdd code-review

tdd — red → green at seams; behavior through public interfaces. code-review — Standards (repo conventions + smell baseline) and Spec (originating issue) in parallel.
