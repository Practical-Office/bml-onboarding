To spec

Turn conversation context into a published spec (PRD) on the issue tracker — synthesize what you already know; do not re-interview.

Why

Durable build contract

After grill-with-docs, the understanding lives in chat unless you capture it. to-spec freezes problem, solution, user stories, implementation decisions, testing decisions, and scope into a tracker issue agents and humans can execute against.

It also forces an explicit conversation about test seams — the public boundaries where behavior will be verified — before to-tickets or implement begin.

When

Use this skill

  • After grill-with-docs on a non-trivial Build experiment
  • When the experiment needs multiple PRs or layered work
  • Before to-tickets splits work

Skip when the build is a single tiny PR — go to implement.

How

Steps

  1. Explore the repo — current code state, domain glossary, ADRs in the touched area.
  2. Sketch test seams — prefer existing seams; propose new ones at the highest practical boundary; aim for as few seams as possible (ideally one). Confirm with the human.
  3. Write the spec using the standard sections: Problem Statement, Solution, User Stories (long numbered list), Implementation Decisions, Testing Decisions, Out of Scope, Further Notes.
  4. Avoid stale detail — no file paths or code snippets unless a prototype encodes a decision more precisely than prose (trim to decision-rich parts).
  5. Publish — create the issue on the project tracker; apply ready-for-agent triage label.

Inputs / outputs

InputsOutputs
Grilled conversation; experiment issue; repo context; confirmed test seams Published spec issue with problem/solution/stories/decisions/testing/out-of-scope; ready-for-agent label; seam agreement for downstream tdd

Anti-patterns

What to avoid

  • Re-interviewing — Facts and decisions should come from prior grill; only confirm seams.
  • File-path specs — Paths rot; describe modules and interfaces instead.
  • Thin user stories — Three stories for a large feature means tickets will guess scope.
  • Skipping seam check — Tests will attach to random internals later.
  • Publishing without ready-for-agent — Downstream agents cannot grab work reliably.

Micro-scenario

Abstract example

After grilling issue #40, the engineer runs /to-spec. The agent reads onboarding modules and CONTEXT.md, then proposes one seam: the public REST endpoint that returns checklist progress.

The human agrees. The spec publishes as issue #41 with fourteen user stories, decisions on event payload shape (no file paths), testing decisions referencing similar API tests elsewhere, and explicit out-of-scope (“no admin editing of checklist items”).

Label ready-for-agent applied. Parent experiment #40 stays open. Next: /to-tickets against #41.
