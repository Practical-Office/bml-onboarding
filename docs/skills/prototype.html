Prototype

Design-time skill · before or beside experiment Build

Throwaway code that answers a question. The question decides the shape — logic/state exploration or UI exploration. Getting the branch wrong wastes the whole prototype.

← Module 5: Running an experiment /implement

Why

Experiment Builds should be thin. If the team cannot yet state how a state machine behaves or which UI layout carries the insight, shipping production code is expensive guessing. A prototype buys a concrete artifact to react to — minutes, not sprints.

Validated decisions fold into the real experiment issue and Build. The prototype itself is captured as a primary source on a throwaway branch; main keeps only the decision.

When

  • Logic branch: “Does this logic / state model feel right?” Hard cases are easier to push through in a tiny interactive terminal app than on paper.
  • UI branch: “What should this look like?” Several radically different UI variations on one route, switchable via URL param and a floating bar.
  • Wayfinder prototype tickets that need something concrete to discuss (HITL)
  • Before /to-spec when the unknown is shape, not requirements volume

Not for production experiment delivery, persistence-by-default systems, or questions already settled by the issue template.

How

1. Pick a branch

QuestionBranchArtifact
Does this logic / state model feel right? Logic Tiny terminal app pushing the state machine through edge cases
What should this look like? UI Multiple UI variants on one route, switchable at runtime

If ambiguous and the user is unreachable: default to logic for backend modules, UI for pages/components — state the assumption at the top of the prototype.

2. Build throwaway (both branches)

  1. Locate code near where it will be used; name it obviously as prototype, not production.
  2. One command to run — pnpm <name>, python <path>, etc.
  3. No persistence by default. State in memory. Scratch DB/file only if persistence is what you’re checking — mark “PROTOTYPE — wipe me.”
  4. Skip polish. No tests, minimal error handling, no abstractions beyond runnable.
  5. Surface state. After every action (logic) or variant switch (UI), show full relevant state.
  6. Capture when done. Fold verdict into real code; commit prototype to throwaway branch; pointer on the issue; record question + answer.

Inputs / outputs

InputsOutputs
Sharp design question (logic or UI) Runnable throwaway artifact marked as prototype
Surrounding module or page context Visible state after each interaction or variant switch
Human reactions during HITL session Verdict recorded on experiment or wayfinder ticket
Project task runner conventions Throwaway branch + context pointer; validated decision in main path

Anti-patterns

  • Wrong branch. Building UI polish when the open question is state transitions (or vice versa).
  • Prototype creep. Adding persistence, tests, and error handling — you’re building the experiment by accident.
  • Anonymous throwaway. Code that looks like production; the next agent ships it verbatim.
  • No capture. Learning dies in chat; issue never records the decision the prototype settled.
  • Prototype instead of Measure. Using throwaway demos to guess customer behavior — that still needs a real experiment and metrics.

Micro-scenario

Logic branch: An experiment will gate “acted on insight” on a three-state flow: delivered → opened → acted. Can a user act without open? Team is unsure.

Prototype: 50-line terminal script. Keys: deliver, open, act, reset. Prints full state after each key. Team discovers acted without opened is allowed — hypothesis table updated before Build.

UI branch: Same experiment needs an in-app insight card. Three layouts: dense table, hero summary, chat-style thread. URL ?variant=1|2|3 + bottom bar. Founder picks variant 2; spec references it; prototype branch linked from issue.
