Why run experiments end-to-end

A ticket that never reaches Learn taught you nothing — no matter how polished the Build was.

Most teams treat the board as a shipping pipeline. Build Loop treats it as a learning pipeline. Each experiment exists to falsify one customer-behavior claim within a fixed duration. The payoff is a recorded decision — persevere, pivot, or kill — backed by numbers, not narrative.

Running one experiment all the way through trains the habits that matter: thin Builds, frozen scope during Measure, and honest Learn outcomes. Skip any column and you get theater — activity that looks like learning but never changes what you build next.

Ideas to hold

  • One issue, one experiment. The GitHub issue is the source of truth; the project board is the visual phase tracker.
  • Build is the only Cursor-heavy column. Agents sharpen plans, publish specs, split tickets, and implement code. They do not decide whether the hypothesis passed.
  • Measure is a clock, not a build phase. Once the thinnest test ships and a collection path exists, the clock starts. No new feature scope until Learn closes the bet.
  • Learn is a decision, not a retrospective vibe. Apply one decision label and file one next action before moving to Done.
  • Kill is a success. Stopping a failed bet early is validated learning, not defeat.

How-to: five columns, one experiment

Backlog Human Create issue from template. Write hypothesis, pass/kill thresholds, duration, sample size. Hypothesis + numeric kill criteria + duration written
Build Human + Cursor Ship the smallest test that lets Measure start. Use the skill chain from Module 4 when non-trivial. Delivery path exists and instrumentation or manual log path is named
Measure Human Collect weekly numbers into the issue. No code unless instrumentation is broken. Duration complete; metrics recorded on the issue

Build column — Cursor workflow

Invoke skills only while the card is in Build. See Module 4 for the full chain and when to skip steps.

  1. /grill-with-docs — prime with the issue + relevant folders; sharpen assumptions and kill criteria
  2. /to-spec — publish a spec from the grilled conversation (skip if the Build fits one tiny PR)
  3. /to-tickets — split into vertical-slice tickets with blockers (skip if one ticket)
  4. /implement — one unblocked ticket at a time; runs /tdd at agreed seams, then /code-review, then commits

For design uncertainty before Build, use /prototype — throwaway code that answers “does this logic feel right?” or “what should this look?”

Agent boundary: Cursor helps on Build. Humans own Measure discipline and Learn decisions. An agent must not apply decision labels or declare a hypothesis validated.

Micro-scenario: weekly insight engagement test

Abstract pattern — swap the nouns for your product.

Hypothesis: Active customers will read and act on a recurring insight delivered each week.

Pass: ≥60% of sampled customers open the insight and complete at least one attributed action per week.

Kill: <40% engagement after four weeks.

Duration: 4 weeks · Sample: 20 active accounts

Backlog: Issue created from template. Measure table filled. Added to board.

Build: Ship email + in-app card for one insight type. Instrument delivered, opened, acted events (or a manual sheet with the same IDs). Cursor runs grill → spec → tickets → implement for the delivery path.

Measure: Every Monday, post open rate and act rate in the issue comment. No “while we’re at it” feature work.

Learn: Week 4 numbers land at 52% act rate — between pass and kill. Team decides pivot: shorten insight, add in-app-only channel, new 3-week experiment filed.

Failure modes

  • Build forever. Polishing the experiment until the duration is half gone. Gate: move to Measure when the thinnest path + collection method exist.
  • Measure creep. “Quick fix” during Measure that changes what you’re testing. File a new experiment instead.
  • Learn by vibes. Moving to Done with “customers seem happy” and no numbers on the issue.
  • Agent decides Learn. Letting Cursor apply persevere because the code merged. Humans compare metrics to the pass/kill table.
  • Vanity metrics. Counting logins or page views when the hypothesis is about action. Prefer outcome events defined before Build.
  • Skipping grill on risky bets. Building before kill criteria are sharp — you learn too slowly to decide.

Module 5 — self-check

  1. Why grill before build on non-trivial experiments?
  2. What triggers moving from Build to Measure?
  3. Who applies the kill-candidate label — and on what evidence?
  4. When is it appropriate to use /prototype instead of jumping to /implement?

Answer in your own words before expanding the answers below.

Show answers
  1. Experiment quality depends on crisp assumptions and a thin Build. Grilling surfaces missing kill criteria, vanity metrics, and scope that would make learning too slow — before /to-spec and tickets.
  2. The thinnest test is shipped and a measurement path exists (instrumentation or a named manual log). Then the clock starts; new feature scope waits for a new experiment.
  3. A human, after duration completes, comparing recorded metrics to the pass/kill table in the issue. The agent may help summarize numbers; it does not apply decision labels.
  4. When the key uncertainty is design shape — “does this state model feel right?” or “what should this UI look?” — before committing to a production Build. Prototypes are throwaway; validated decisions fold into the real experiment issue.
