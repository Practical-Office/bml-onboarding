Introduction

~10 min

Why BML (not a feature factory)

The goal is not to ship features faster. The goal is to run disciplined experiments that test whether customers actually get value.

New products are built under uncertainty. The hard questions are “Should we build this?” and “Will customers sustain value from it?” — not only “Can we ship it?”

Validated learning is the unit of progress: you empirically show you learned something true about customers or the business by running a real-world experiment. Shipping volume without learning is activity theater.

Concepts such as Build–Measure–Learn, validated learning, MVP, and pivot/persevere come from Eric Ries’s Lean Startup methodology (theleanstartup.com/principles). This course teaches those ideas at a high level and does not reproduce book text. Practical AI’s Kanban adds an explicit kill outcome alongside pivot and persevere.

Risky assumptions you must name

Before you open a ticket, name the leap-of-faith assumptions the experiment will test. Examples of the shape (swap for your product):

Assumption typeWhy it’s risky
Customers act on the core insight / recommendationCore value hypothesis
A differentiator (human review, unique data, etc.) actually changes trust or usagePositioning bet
Customers renew after novelty wears offRevenue survival
A roadmap feature is among the most usedBuild priority bet

BookIQ teammates: see concrete product assumptions in the appendix.

Expected outcomes

After this course you will:

  • Create a BML experiment issue in under 2 minutes
  • Move tickets through Build → Measure → Learn with clear kill criteria
  • Use Cursor skills on Build without skipping hypothesis work
  • Submit a sandbox experiment for lead review

Module 1 — progress checklist

Module 1 — self-check

  1. What is the difference between shipping features and running BML experiments?
  2. What happens if Learn never results in killing or pivoting anything?
Show answers
  1. Shipping features optimizes for delivery. BML experiments optimize for validated learning: each ticket states a falsifiable customer-behavior hypothesis, defines pass/kill thresholds up front, and ends with a decision grounded in evidence.
  2. You are doing theater. If Learn rarely produces persevere / pivot / kill, the loop is broken — stop shipping new experiments until decisions catch up.

The BML Loop

~15 min

The loop

Turn ideas into the smallest testable product, measure how customers respond, and learn whether to change course. Optimize for faster cycles through the loop — not bigger Builds.

     ┌─────────┐
     │  BUILD  │  Ship the smallest test (MVP for learning)
     └────┬────┘
          ▼
     ┌─────────┐
     │ MEASURE │  Collect data against pass/kill thresholds
     └────┬────┘
          ▼
     ┌─────────┐
     │  LEARN  │  Persevere · Pivot · Kill
     └────┬────┘
          │
          └──────► next experiment

Phases (what each means)

PhaseYou are…Exit criteria
BuildShipping the thinnest test that starts learning (MVP as learning vehicle, not “half a roadmap”)Instrumentation or manual log path exists
MeasureClock running; no new scope; prefer actionable metricsDuration complete; numbers recorded on the issue
LearnDeciding with evidenceLabel: persevere / pivot / kill-candidate

Actionable vs vanity metrics

Prefer metrics that support a decision and show cause–effect. Avoid vanity totals that look good but do not change what you do next (raw logins, undifferentiated click counts).

Light “innovation accounting”: set a baseline from the first MVP, tune toward a target, then decide. Pass/kill rows on the ticket are enough for day-to-day ops — see BML Metrics Tracking.

Learn outcomes

OutcomeMeaning
PersevereLearning supports the current strategy — continue or expand
PivotStructured course correction: change hypothesis/approach while keeping vision; open a revised experiment
KillPractical AI extension: stop the experiment / drop the bet when criteria fail — explicit third outcome so theater cannot continue

Pitfalls (hard truths)

If Learn does not regularly kill, change, or pivot features, you are doing theater, not BML.

  • Vanity metrics — activity without “acted on value” signal
  • Skipping Measure — deciding on vibes before the clock runs
  • Building the full feature — experiment scope too large to learn fast
  • No kill criteria — hypothesis unfalsifiable

Module 2 — progress checklist

Module 2 — self-check

  1. Why is Measure a separate phase instead of “check analytics whenever”?
  2. What makes an experiment falsifiable?
Show answers
  1. Measure is a gated phase with a clock and no new scope. Separating it prevents “we’ll just peek at dashboards” from collapsing into Build forever — and forces numbers onto the issue before Learn.
  2. You define numeric pass and kill thresholds (and a duration) before Build starts. If the hypothesis cannot fail, it is not an experiment.

GitHub Setup

~15 min

What you need

PieceRole
GitHub IssuesOne issue = one experiment (source of truth)
GitHub ProjectVisual Kanban (columns = phase)
Label experimentMarks every BML ticket
Labels persevere / pivot / kill-candidateLearn decision only

Live rule: We do not use hypothesis / measure / learn labels. Columns own the phase. Extra phase labels create drift and slow ticket creation.

Step-by-step

  1. Create labels (once per repo): 
    gh label create experiment --description "BML experiment ticket" --color 0E8A16
gh label create persevere --description "Learn decision — keep going" --color 0E8A16
gh label create pivot --description "Learn decision — change approach" --color FBCA04
gh label create kill-candidate --description "Learn decision — kill or sunset" --color D93F0B
    Verify with gh label list | rg 'experiment|persevere|pivot|kill-candidate'.
  2. Create or open a Project board with stages: Backlog · Build · Measure · Learn · Done. Recreate tooling: bash scripts/create-bml-project.sh (needs gh auth refresh -s project,read:project).
  3. Wire the issue template — teaching copy: bml-experiment.md. Live product templates stay on the product repo (BookIQ: see appendix).
  4. Group the board by Stage — one-time UI click: Board view → Group by → Stage. Filter tip: label:experiment.

Detailed setup: BML Kanban Setup

Module 3 — progress checklist

Module 3 — self-check

  1. Why don’t we use phase labels when we have phase columns?
  2. What is the source of truth — the board or the issue?
Show answers
  1. Columns already own the phase. Phase labels duplicate state, drift out of sync with the board, and slow creation. Labels are reserved for type (experiment) and Learn decisions.
  2. The issue is source of truth (hypothesis, metrics, evidence, decision). The Project board is the visual Kanban only.

Using the BML Ticket Template

~10 min

Full template

## Hypothesis
One sentence. What customer behavior do we believe will happen?

## Build (smallest test)
What is the thinnest thing we ship to test this?

## Measure
| Metric | Target (pass) | Kill (fail) | How collected |
|--------|---------------|-------------|----------------|
|        |               |             |                |

**Duration:** ___ days / weeks
**Sample:** ___ customers / tenants

## Learn (fill only when deciding)
- **Decision:** Persevere / Pivot / Kill
- **Evidence:** (numbers + quotes)
- **Next action:** (one concrete follow-up)

## Acceptance Criteria
- [ ] Hypothesis + kill criteria before Build
- [ ] Instrumentation before Measure
- [ ] Learn decision with evidence
- [ ] Decision label applied

How to fill each section

SectionRule
HypothesisOne sentence; falsifiable customer behavior (not “users will like it”)
BuildSmallest test that starts learning — not the full feature
MeasurePass and kill thresholds; duration; how collected (event IDs or manual sheet)
LearnFill only at decision time — numbers and next action, not vibes

Ready to leave Backlog: one-sentence hypothesis, numeric kill criteria, duration set, measure path named.

Metrics dictionary: BML Metrics Tracking · Teaching template: bml-experiment.md

Module 4 — progress checklist

Module 4 — self-check

  1. What’s wrong with a hypothesis that says “users will like the feature”?
  2. When do you apply kill-candidate?
Show answers
  1. It is not falsifiable or behavioral. “Like” has no measurable pass/kill path. Rewrite as observable customer behavior with thresholds (e.g. open rate, act-on-insight rate, return within N days).
  2. At Learn, when evidence hits (or clearly fails) the kill threshold — or when the team decides the bet should stop. Apply the label, record evidence, and file one next action.

Creating & Running Your First Experiment

~25 min

Walkthrough: Core Value Test (portable pattern)

Hypothesis: Customers will actively read and act on weekly insights.
Pass: ≥60% open + click through weekly. Kill: <40% after 4 weeks.

This is the pattern for any core-value experiment: name the behavior, set pass/kill, set duration, ship the thinnest delivery path that lets Measure start.

Steps:

  1. Create issue from template → label experiment (pre-set on the template)
  2. Add to board → Backlog
  3. Move to Build only when hypothesis + kill criteria + duration are written
  4. Cursor workflow (Build only):
    • /grill-with-docs — prime with the issue + relevant folders (skip if Build is a tiny slice)
    • Spec only if non-trivial
    • /to-issues or implement directly if the Build fits in one PR
    • /tdd → /code-review
  5. Move to Measure — clock starts; no new scope; update the issue with weekly numbers
  6. Move to Learn — record decision + apply one decision label + next action

Agent note: Cursor helps on Build. Humans own Measure discipline and Learn decisions.

BookIQ week-1 drafts (Core Value, Onboarding, Feature usage) live under reference/issues/ and are linked from the appendix.

Module 5 — progress checklist

Module 5 — self-check

  1. Why grill before build on non-trivial experiments?
  2. What triggers moving from Build to Measure?
Show answers
  1. Experiment quality depends on crisp assumptions and a thin Build. Grilling surfaces missing kill criteria, vanity metrics, and scope that would make learning too slow — before you spend engineering time.
  2. The thinnest test is shipped and a measurement path exists (instrumentation or a named manual log). Then the clock starts; new feature scope waits for a new experiment.

Operating the Board

~15 min

Moving tickets

From → ToGate
Backlog → BuildHypothesis + kill criteria written
Build → MeasureInstrumentation or manual log exists
Measure → LearnDuration complete; numbers in issue
Learn → DoneDecision label + next action filed

Measurement discipline

  • Max 3 active experiments in Build + Measure combined (small team)
  • Week-1 Measure can be a spreadsheet — event IDs defined in Metrics doc so later instrumentation does not invent new names
  • Record numbers in the issue body at Learn — the board card is not enough

Learn-phase decisions

LabelMeaning
persevereHypothesis supported — continue / expand
pivotSignal mixed — change approach, new experiment
kill-candidateHypothesis failed — sunset or stop

Weekly cadence (≈30 min)

WhenWhat
MondayReview board; unblock Measure; move anything past duration into Learn
WednesdayCheck active experiments still in scope (no silent Build creep)
FridayAnything in Learn gets a decision or an explicit defer with a date

If Learn has zero decisions for 2+ weeks, fix that before shipping more features. Ops detail: Setup · Implementation Notes.

Module 6 — progress checklist

Module 6 — self-check

  1. Can you move to Learn without kill/pass numbers? Why not?
  2. What do you do when an experiment is in Learn for two weeks with no decision?
Show answers
  1. No. Learn without numbers is vibes. Duration must be complete and metrics recorded on the issue so the decision is auditable.
  2. Treat it as a system failure: force a decision in the next weekly review (persevere / pivot / kill), or explicitly defer with a date. Do not open new Builds until Learn clears.

Your First Project

~30 min

Mandatory exercise

Create your real first BML issue — as a sandbox. Do not add it to the live board until a lead graduates it. Practice must not pollute the operating board.

Requirements:

  1. Use the experiment template completely
  2. Hypothesis + pass/kill metrics + duration filled
  3. Label experiment applied
  4. Issue stays draft/sandbox until lead promotes it

Submission

Post your issue URL where your lead specifies (repo Discussion, Slack, or team channel):

BML Module 7 submission
Issue: https://github.com/YOUR-ORG/YOUR-REPO/issues/NNN
Hypothesis (one line): …

BookIQ teammates: create the sandbox issue on Book-IQ/bookiqv1-rc; lead graduates it to the BML Experiments board when the template and thresholds are sound.

Module 7 — progress checklist

Module 7 — self-check

  1. Why sandbox before adding to the live board?
  2. What must a lead check before graduating your issue?
Show answers
  1. So practice tickets do not crowd the live WIP limit or dilute active experiments. Leads keep the operating board honest.
  2. Complete template, falsifiable hypothesis, numeric pass/kill + duration, measure path named, and that the experiment fits under the 3-active Build+Measure cap when promoted.
