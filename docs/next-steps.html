2 · Homework F

One live experiment through Build

Book-IQ members: within the next sprint (or agreed window), complete one self-chosen live experiment from the BML backlog (lead approves your pick). Non–Book-IQ: run the same Homework F on your org’s experiment board with lead sign-off.

  1. Pick + approve — choose a live backlog card (see teaching drafts as shape examples, not copy-paste sources). Lead confirms WIP ≤3 and sequencing (e.g. no Pricing before Core Value signal).
  2. Sharpen the live issue — six-section template on the real GitHub issue; /grill-with-docs if hypothesis or Technical Context is vague.
  3. Backlog → Build — apply experiment label; move only after hypothesis + numeric kill criteria are written.
  4. Build skill chain — /grill-with-docs → /to-spec → /to-tickets → /implement (tiny builds may skip to /implement).
  5. Ship smallest test — homework gate: Build started or first PR merged on the live ticket.
  6. Stretch: Measure — move to Measure and post week-0 baseline numbers on the issue.

Not homework gate: full Learn decision — that is ongoing board ops once the experiment duration elapses.

3 · Operate the board

Weekly rhythm + WIP

  • Max 3 active cards in Build + Measure combined.
  • Weekly Learn review — decisions before new Builds.
  • If Learn produces no persevere / pivot / kill for 2+ weeks, fix the system before shipping more scope.
  • Post weekly Measure numbers on the issue; no new product scope in Measure unless instrumentation is broken.

Homework failure modes

  • Moving a live card without lead-approved pick.
  • Copy-pasting a teaching draft onto the live issue without sharpening for the real bet.
  • Measure section without numeric kill threshold.
  • Vanity metrics (raw logins, undifferentiated clicks) as success criteria.
  • Skipping Measure or Learn — parking cards with no decision label.
  • WIP >3 — starting homework before finishing or parking other active experiments.
