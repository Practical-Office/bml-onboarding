Ticket Template & Build Skills

The experiment issue template forces hypothesis-before-build discipline. In the Build column, a four-skill chain turns that issue into shipped, tested code — one tracer-bullet ticket at a time.

4

Why

Same shape every time; skills for non-trivial Build

~25 minutes

Experiments fail quietly when the ticket is vague: no kill criteria, no duration, no measure path. A fixed template makes the minimum bar obvious before anyone writes code.

Non-trivial Build work needs more than a template. The skill chain — grill-with-docs, to-spec, to-tickets, implement — sharpens the plan, publishes a spec, slices vertical tickets with blocking edges, then builds one ticket at a time with test-driven development and code review. Tiny builds can skip straight to implement.

Do not substitute to-issues plus bare tdd for this chain. implement owns tdd and code-review at the end of each ticket.

How-to

File and run an experiment build

1. Create the issue (<2 minutes)

  1. New issue → template Build Loop Experiment
  2. Fill Hypothesis, Build, Measure table (pass/kill), Duration
  3. Label experiment is pre-set; add card to project Backlog

2. Move to Build — pick a path

Tiny build (single small PR): optional light grill-with-docs, then implement directly.

Non-trivial build:

  1. /grill-with-docs on the issue — prime folders, sharpen plan, capture glossary/ADRs
  2. /to-spec — synthesize conversation into a published spec; confirm test seams
  3. /to-tickets — vertical slices with blockers; publish; work the frontier
  4. /implement — one ticket at a time; clear context between tickets

3. Measure and Learn (no skill chain)

  • Measure: update the issue with weekly numbers. Code only if instrumentation is broken.
  • Learn: human decision; optional grill-with-docs if pivoting. Apply decision label; move to Done.
4

Micro-scenario

Abstract walkthrough

Context: Issue #40 proposes a guided checklist inside an onboarding flow. Build is multi-step (API + UI + analytics).

  1. PM files the template with hypothesis “users who see the checklist complete setup 20% more often,” kill at <5% lift, two-week measure window.
  2. Engineer runs grill-with-docs: one question at a time until “checklist item” vs “setup step” terms land in CONTEXT.md.
  3. to-spec publishes a spec: user stories, API contract decisions, test seams at the public onboarding API.
  4. to-tickets yields three tracer bullets: (1) API + empty UI shell, (2) checklist render + event, (3) dashboard readout — each blocked only by its predecessor.
  5. implement on ticket 1: tdd at the API seam, full suite once, code-review, commit. Context cleared; ticket 2 next.
  6. Card moves to Measure; no further skills until numbers arrive.

Failure modes

  • Full chain for a one-line fix — Adding analytics to an existing button does not need a spec and five tickets. Use implement.
  • Skipping grill on ambiguous scope — “Improve onboarding” without shared terms produces specs that rewrite themselves mid-build.
  • Horizontal tickets — “All API work” then “All UI work” cannot demo mid-flight. Use vertical tracer bullets.
  • Implementing multiple tickets in one context — Context rots; blockers blur. One ticket, then clear.
  • Bare tdd without implement’s review — Tests land without standards/spec check. Let implement run the full close-out.
  • Measure-phase feature creep — “While we wait, let’s also…” Invalidates the experiment clock.

Try answering aloud before opening the answers.

Answers
  1. Hypothesis (one sentence), Build (smallest test), Measure (pass/kill table + duration/sample), and acceptance that kill criteria exist before build — Learn stays empty until decision time.
  2. grill-with-docs → to-spec → to-tickets → implement.
  3. When the build fits in a single small PR (e.g. tiny instrumentation fix); optional light grill first.
  4. tdd (at pre-agreed seams) and code-review (standards + spec axes).
  5. A vertical slice that cuts a narrow but complete path through every layer, demoable on its own, sized for one fresh context window.
